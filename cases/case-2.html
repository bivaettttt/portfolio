Интеграция CRM ↔ сервис уведомлений

Кейс про интеграцию через REST API: контракт, статусы, ошибки, ретраи, идемпотентность и критерии приемки.

Цель

Обеспечить отправку уведомлений из CRM через отдельный сервис с предсказуемыми статусами, ошибками и правилами повторных попыток.

Эффект для бизнеса

Прозрачная доставка и аудит событий, меньше потерь уведомлений, контролируемые ретраи и единый контракт для поддержки/развития.

Моя роль

  • Собрал требования и зафиксировал границы скоупа
  • Описал сущности, статусы и основные сценарии
  • Сформировал контракт REST API (OpenAPI) + коды ошибок

  • Подготовил acceptance criteria и описание UML sequence

Контекст

CRM должна отправлять SMS/Email/Push при событиях (создание заявки, смена статуса, просрочка SLA). Вариант “вшить отправку в CRM” усложняет поддержку и делает доставку менее предсказуемой. Решение — выделить отправку в отдельный сервис уведомлений и интегрировать через REST API.

Скоуп

Входит

Создание уведомлений по API, получение статуса доставки, обработка ошибок, логирование, правила ретраев.

Не входит

UI для шаблонов, биллинг сообщений, выбор провайдера по стоимости, админка провайдеров.

Роли и сущности

  • CRM
    источник события, инициатор отправки
  • Notification Service
    валидирует, ставит в очередь, отправляет, трекает
  • Provider
    внешний SMS/Email/Push шлюз
  • Notification
    объект отправки с id, статусом, payload

Контракт API (структура)

POST /api/v1/notifications
Создать уведомление

Заголовки: Idempotency-Key, X-Correlation-Id.
Тело: канал (sms/email/push), получатель, шаблон/текст, параметры, приоритет.

GET /api/v1/notifications/{id}
Получить статус

Статусы: NEW → QUEUED → SENT → DELIVERED / FAILED.
Дополнительно: attempts, lastError, providerMessageId.

Ошибки и ретраи
предсказуемые коды

400: неверный payload / отсутствуют поля.
401/403: авторизация/права.
409: конфликт идемпотентности (ключ уже использован с другим payload).
429: rate limit (ретрай с backoff).
5xx: временная ошибка сервиса/провайдера (ретрай по политике).

UML sequence (описание)

1) CRM → Notification Service: POST createNotification (Idempotency-Key).
2) Service валидирует → пишет запись NEW → ставит в очередь → возвращает 202 + id.
3) Worker → Provider: sendMessage → получает providerMessageId.
4) Service обновляет статус SENT, затем DELIVERED/FAILED по callback/опросу провайдера.
5) CRM → Service: GET status → получает текущий статус и историю попыток.

Критерии приемки (пример)

  • AC-01
    При валидном запросе сервис возвращает 202 + notificationId
  • AC-02
    Повтор с тем же Idempotency-Key возвращает тот же notificationId
  • AC-03
    GET /{id} возвращает статус и attempts
  • AC-04
    При 429/5xx выполняются ретраи по политике (фиксируется attempts)
