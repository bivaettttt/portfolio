Интеграция CRM ↔ сервис уведомлений

Учебный кейс в формате реального проекта: проработка требований, контракта API, ошибок, ретраев и критериев приемки.

Цель

Обеспечить отправку уведомлений клиентам из CRM через внешний сервис с прогнозируемыми статусами, ошибками и повторными попытками.

Результат

Единый контракт API, фиксированные коды ошибок и правила ретраев, прозрачный трекинг доставки и аудит событий.

Контекст

CRM должна отправлять SMS/Email/Push при событиях (создание заявки, смена статуса, просрочка SLA). Ранее отправка была “вшита” в CRM → сложная поддержка и нестабильная доставка. Решение — вынести отправку в отдельный сервис уведомлений и интегрировать через REST API.

Скоуп

Входит

Отправка уведомлений по API, получение статуса доставки, обработка ошибок, логирование, правила ретраев.

Не входит

UI для шаблонов, биллинг сообщений, маршрутизация каналов по стоимости, админка провайдеров.

Роли и сущности

  • CRM
    источник события, инициатор отправки
  • Notification Service
    валидирует, ставит в очередь, отправляет, трекает
  • Provider
    внешний SMS/Email/Push шлюз
  • Notification
    объект отправки с id, статусом, payload

Требования

Функциональные

1) CRM отправляет запрос на создание уведомления.
2) Сервис возвращает идентификатор уведомления.
3) CRM может запросить статус доставки.
4) Сервис обрабатывает ошибки провайдера и сохраняет историю попыток.

Нефункциональные (базово)

Идемпотентность на уровне запроса, таймауты, ретраи по временным ошибкам, аудит и трассировка (correlationId).

Контракт API (черновик)

POST /api/v1/notifications
Создать уведомление

Заголовки: Idempotency-Key, X-Correlation-Id.
Тело: канал (sms/email/push), получатель, шаблон/текст, параметры, приоритет.

GET /api/v1/notifications/{id}
Получить статус

Статусы: NEW → QUEUED → SENT → DELIVERED / FAILED.
Дополнительно: attempts, lastError, providerMessageId.

Ошибки и правила ретраев
Согласованные коды

400: неверный payload / отсутствуют обязательные поля.
401/403: авторизация/права.
409: конфликт идемпотентности (если ключ уже использован с другим payload).
429: rate limit (ретрай с backoff).
5xx: временная ошибка сервиса/провайдера (ретрай по политике).

UML sequence (описание)

1) CRM → Notification Service: POST createNotification (Idempotency-Key).
2) Service валидирует → пишет запись NEW → ставит в очередь → возвращает 202 + id.
3) Worker → Provider: sendMessage → получает providerMessageId.
4) Service обновляет статус SENT, затем DELIVERED/FAILED по callback/опросу провайдера.
5) CRM → Service: GET status → получает текущий статус и историю попыток.

Критерии приемки (пример)

  • AC-01
    При валидном запросе сервис возвращает 202 + notificationId
  • AC-02
    При повторе с тем же Idempotency-Key возвращается тот же notificationId
  • AC-03
    GET /{id} возвращает статус и attempts
  • AC-04
    При 429/5xx выполняются ретраи по политике (фиксируется attempts)
